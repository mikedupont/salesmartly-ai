# salesmartly-ai 项目计划

这份计划跟着当前代码进度走，目标是让后续开发有清晰优先级，不把功能堆成一团。

## 当前阶段

当前项目处于“可持续迭代的工程底座”阶段，已经完成了：

- SaleSmartly 消息接入
- 长期记忆主链路
- D1 结构化存储
- Cloudflare Vectorize 语义记忆
- Relationship State 关系状态机
- Dialogue Policy Engine 对话策略层
- Post-processing 回复收敛层
- Memory Writer 写回编排层
- `/admin` 记忆查看页
- `/admin/memory` 调试接口
- `/health` 健康检查
- 基础烟雾测试
- 增量摘要链路
- 训练样本采集与导出链路
- 偏好反馈写回链路
- 训练样本后台标注链路
- 身份追问安全分支与后处理拦截
- `status=unlabeled` 的未标注工作流
- FlirtFlip 风格种子生成与清洗链路
- 当前可直接训练的公开对话清洗版
  - FlirtFlip seed SFT：2142
  - FlirtFlip seed DPO：2142
  - FlirtFlip final SFT：1071
  - FlirtFlip final DPO：1071

## 里程碑 1

目标：把“能看见、能排查、能定位”的基础先补齐。

已完成或正在完成：

- 记忆查看接口
- 记忆调试接口
- 调试页展示策略层结果
- 调试页展示事实、摘要、向量、关系状态
- 训练样本查看与导出
- 偏好反馈记录

下一步：

- 继续补全后台页信息密度
- 增加更直观的调试视图
- 补充训练数据质量统计

## 里程碑 2

目标：把向量记忆做得更稳定、更准。

重点方向：

- 检索排序优化
- 相似结果去重
- 召回阈值调优
- 不同记忆源权重区分
- 更好的 query probe 组合

## 里程碑 3

目标：把后台做成真正可用的运维界面。

重点方向：

- 用户维度总览
- 关系状态展示
- 事实记忆编辑/查看
- 摘要历史查看
- 向量召回结果查看
- 单用户对话轨迹查看

## 里程碑 4

目标：把质量保障补起来。

重点方向：

- 更完整的测试覆盖
- 冒烟测试扩展
- 监控快照增强
- 常见异常告警信息
- 线上问题定位路径

## 后续增强方向

在基础稳定后，再继续做：

- 更自然的对话节奏
- 更细的陪伴风格控制
- 更稳定的长期关系演化
- 更稳定的身份追问处理，不在普通聊天中自曝系统实现
- 更好的 Worker 职责切分，把后台和训练路由继续下沉
- 更完整的多用户隔离
- 更适合商业化的权限和后台

## 里程碑 5

目标：把模型本身也调成更像真人，而不是只靠 prompt 和后处理。

推荐路线：

- 收集真实偏好样本
  - 同一输入保留多条候选回复
  - 标出更自然、更少 AI 味的版本

- 轻量监督微调
  - 用 LoRA / PEFT 做小成本风格适配
  - 先把基础聊天语气调顺

- 偏好优化
  - 用 DPO 或 RLHF 继续把“像人”的回复往上推
  - 把客服味、模板味、追问过多压下去

- 推理期兜底
  - 继续保留 Dialogue Policy Engine
  - 继续保留 Post-processing
  - 继续保留身份追问专用安全分支
  - 让训练和策略双保险配合

这一步完成后，项目会从“规则和 prompt 驱动的陪聊系统”进化成“经过偏好训练的长期陪伴 AI”。

## 里程碑 6

目标：真正形成一个能不断长出训练资产的闭环。

重点方向：

- 自动导出训练数据集
- 偏好标签审核与回写
- 后台一键标注 chosen / rejected
- 只看未标注样本的连续标注模式
- 样本质量统计
- 好 / 坏回复对比分析
- 为 LoRA / DPO 准备干净数据
- 训练实现细节见 [`TRAINING_IMPLEMENTATION.md`](TRAINING_IMPLEMENTATION.md)
- 当前公开对话清洗版已经生成，可直接拿去继续整理训练集

## 推荐开发顺序

1. 先补记忆查看和调试接口
2. 再优化向量召回
3. 再完善后台页面
4. 最后补测试和监控

## 当前判断

这套系统已经不是“聊天机器人 demo”了，而是一个可继续长出来的长期陪伴 AI 底座。
